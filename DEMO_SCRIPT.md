1. **啟動**：
   ```bash
   cd ~/Projects/xiaoxiao-yibiaotong
   source .venv/bin/activate
   streamlit run app.py --server.port 8501
   ```
2. **環境檢查**：sidebar 應顯示

## ⚡ 3 分鐘 Pitch（評審 / 校長遴選短講）

> **目標**：建立「我用校長視角 + 完整願景 + 學術依據」三件套印象

### 🎬 流程

| 秒數 | 動作 | 講詞重點 |
|------|------|---------|
| 0:00-0:20 | 開校長儀表板（首頁），指 hero | 「**校長不孤單。13 條管考線，13 個 AI 副手分擔。**校長一個人面對採購、公文、校事、輔導、特教、課程… 每條都要 sign-off，但每條都不可能精通。」 |
| 0:20-0:50 | 指 4 metric cards | 「校長進入系統，只看四件事：**今天要核決幾件、有幾件高風險、主任已經處理多少、這週新進案件**。其他細節 AI 副手分擔。」 |
| 0:50-1:20 | 滾到 13 子系統溫度計 | 「13 子系統用紅黃綠標。一眼看到 🔴 S03 採購 45 萬平板簽呈待我核 → 這就是我今天該介入的。其他 🟢 的我不用管。」 |
| 1:20-1:50 | 滾到 7 層視角覆蓋率 | 「整個系統覆蓋校長、主計、總務主任、教導主任、兩組長、承辦人 7 層。每層只看自己該看的，**校長不接管承辦人工作**。」 |
| 1:50-2:20 | 切到 🏛️ 系統架構 tab，滾「整體架構」ASCII | 「願景：13 子系統、3 共用模組。今天 demo 的 web app 只覆蓋 3/13（採購 + 公文 + 會議），其他 10 條規劃中。**這是路線圖，不是終點。**」 |
| 2:20-2:50 | 切「學術 USP」段 | 「Stanford 2025 證實商用法律 AI 工具有 17-33% 錯誤率 — 我的設計是 **自建 RAG + 人在環中**。USP 四維：小校 × 總務 × 中文法規 × Tool use，SSCI 文獻真空地帶。」 |

## 🎯 10 分鐘 Demo（教師研習 / 同事介紹 / 比賽答詢）

> **目標**：讓對方相信「明天就能用」+「真的會省時間」

### 🎬 流程

#### Part 1：故事開場（0-2 分鐘）
- 開校長儀表板（首頁），指 hero：「**校長不孤單。13 條管考線，13 個 AI 副手分擔。**」
- 指 4 metric cards：「校長視角只看四件事 — 待我核決、有風險、主任已處理、本週新進」
- 滾 13 子系統溫度計：「一眼看到🔴 S03 採購 45 萬平板簽呈 — 這就是我今天該介入的」
- 講具體痛點：「過去校長要核 45 萬平板採購，得自己查《政府採購法》三條條文。現在 AI 副手已經幫總務主任查過、寫好簽呈、標好風險，校長只看『要不要核』」

#### Part 2：模組 A 採購法顧問（2-5 分鐘）
- 切到 📋 採購法顧問 tab
- **Demo 模式**：點「⚡ 45 萬平板採購簽呈審查」
  - 講解：條號引用、白話解釋、學校場景、風險提醒、建議做法
  - 強調：「最後那句『實際採購決策請以機關法務、主計與主管核示為準』 — **AI 不會逾越，把決定權還給人**」
- **Live 模式**（若有 API key）：勾選「🔬 RAG 模式」，自由問一題如「公告金額未滿 10 萬可不可以分次採購」
  - 觀察：spinner 顯示「查詢真實條文中...」
  - 結果有「✅ RAG 模式：已查詢採購法彙編（35 版 684 頁）」綠色提示

#### Part 3：模組 B 公文草稿（5-7 分鐘）
- 切到 📝 公文草稿 tab
- 點「⚡ 申請操場跑道整修補助」
  - 講解：三段式（主旨 / 說明 / 辦法）自動產出
  - 強調：「總務主任過去寫一份公文 15 分鐘，現在 30 秒，**人去做議價、議價是不能 AI 化的，要面對廠商**」
- 自由 demo：填寫「事由：申請夜光跑道補助 / 受文者：教育處 / 類型：申請」+ 關鍵事實 3 點，按「✍️ 生成公文草稿」

#### Part 4：模組 C 會議記錄（7-9 分鐘）
- 切到 🎙️ 會議記錄 tab
- 點「⚡ 1-click 示範」載入逐字稿
- 點「📝 生成會議記錄」
  - 結果：摘要 + 決議事項 + 待辦清單三段式
- **本機模式才有**：滾到下方，按「📅 把決議寫進 macOS 行事曆」
  - 觀察：Agent 抽取每筆決議 → 呼叫 add_to_calendar → macOS 行事曆真的跳出新事件
  - 強調：「這就是『**Tool use**』— AI 不只回答，還動手做事。」

#### Part 5：收尾（9-10 分鐘）
- 切回 🏛️ 系統架構 tab，滾到「學術 USP」
  - 「這套設計有 30 篇 SSCI/arXiv 文獻支撐，準備投 Educational Management Administration & Leadership」
- 切到 🏫 本校資料 tab
  - 「最後：本校資料只填一次，三模組共用 — **這就是『一表通』的核心**」
- 結尾：「歡迎用，歡迎拍磚。Streamlit Cloud 版本：[URL]」

## 🎯 30 分鐘 Deep Dive（共備坊 / 技術討論 / 博論田野）

> **目標**：讓對方理解架構決策、可以自己改、看到未來可能性

### 🎬 流程

#### 第 1 段（0-5 分鐘）：哲學與場域
- 開首頁、講六班小校的結構：總務 = 1 人、會計 = 人事幹事兼任、出納 = 校護兼任
- 引用：「黃秀琴科長的『行政減負，不是稽核』哲學 — 看板用『需要支援』不用『逾期』」
- 提：vault 的 08_校務減負系統 已有 02_主架構 v1.9 + 13 子系統 v2.0 規劃

#### 第 2 段（5-12 分鐘）：三模組逐一深入
- 跟 10 分鐘版本同，但每個模組多花 1-2 分鐘
- 額外講：
  - 模組 A 的 RAG 設計：「為什麼是 grep + 前後 2 行 context，不是 vector search？」答：六班小校沒 GPU、彙編就 750KB、grep 又快又準
  - 模組 B 的 prompt：開 prompts/official_doc.md 給看
  - 模組 C 的 macOS Calendar tool：開 app.py:331-373 給看 osascript 段

#### 第 3 段（12-20 分鐘）：架構決策
- 切到 🏛️ 系統架構 tab，逐一展開所有 expander
- 講：
  - 為什麼分 7 層視角而不是 3 層 → 教務組長/訓導組長分流公文（S02 的設計）
  - 為什麼 S12 是「父」、S11 是「子」 → 計畫框父集合，任務派發為子單元，案號關聯
  - 為什麼小校一表通只覆蓋 3/13 → 邊界明確：web app 給現場使用者、CLI skill 給深度使用者、HTML 原型給 UX 設計

#### 第 4 段（20-25 分鐘）：學術定位
- 切到「學術 USP」段
- 講雙軌文獻調研流程：WoS researcher agent + Semantic Scholar agent 並行
- 「Stanford 2025 那篇打臉商用 LegalAI 工具 17-33% 錯誤 — 我們的差異化是 RAG + 人在環中 + 透明引用條號」
- 「USP 四維齊全 — 小校、總務、中文法規、Tool use — 是真空地帶。投稿首推 EMAL」

#### 第 5 段（25-30 分鐘）：可擴充性 + Q&A
- 開 SUBSYSTEM_MAP.md（在 xiaoxiao-yibiaotong 根目錄）
- 講：「這個 web app 不是終點。S09 防災、S10 經費、S12 計畫案件庫都是天然下一步」
- 開放 Q&A

## 🔧 故障排除

| 問題 | 解法 |
|------|------|
| Streamlit 起不來 | `pip install -r requirements.txt --upgrade` |
| Agent SDK 模式找不到 claude CLI | `npm i -g @anthropic-ai/claude-code` |
| 行事曆寫入失敗 | macOS 系統設定 → 隱私權 → 自動化 → 終端機（或 iTerm）→ 勾選 Calendar |
| RAG 沒命中關鍵字 | 切回普通模式（取消 RAG checkbox），用 LLM 內建知識回答 |
| Demo 模式沒回應 | 確認 tests/demo_cases/ 目錄下三個 .md 都在 |
